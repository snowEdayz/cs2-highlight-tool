# Frontend Directory Structure

> How Vue 3 + TypeScript frontend code is organized.

---

## Overview

The frontend lives entirely under `frontend/src/` and follows a **feature-first** layout: each business domain is a directory under `features/`, with its own pages, components, and composables. Cross-cutting utilities go in `shared/`.

## Feature Directory Convention

Each feature directory contains:

| Subdirectory | Purpose | Required |
|-------------|---------|----------|
| `pages/` | Route-level page components (one per route) | вњ“ (pages\* have at least one) |
| `components/` | Feature-specific reusable components | Optional |
| `composables/` | Composition API state/effect hooks | Optional |

> **Exception**: The `ads` feature has no page вЂ” it's a cross-feature component injected into the startup and main views.

**Imports within a feature**: Use relative imports for files in the same feature, `@/` alias for cross-feature and shared imports.

---

## File Naming

| Type | Convention | Examples |
|------|-----------|---------|
| Vue components | `PascalCase.vue` | `ClipsPage.vue`, `DeathNoticeLine.vue`, `ImportDemoList.vue` |
| Composables | `usePascalCase.ts` | `useStartupWizard.ts`, `useImportDemos.ts`, `useDebugSettings.ts` |
| Shared modules | `camelCase.ts` | `events.ts`, `router.ts` |
| Type domain files | `camelCase.ts` (under `types/` dir) | `startup.ts`, `demo.ts`, `import.ts`, `clips.ts`, `edit.ts` |
| i18n files | `locale-Region.json` | `zh-CN.json`, `en-US.json` |

---

## Import Alias

Use `@/` for paths under `frontend/src/`:

```ts
import { t } from "@/shared/i18n";
import type { DemoClipKill } from "@/shared/types";
import { useImportDemos } from "@/features/import/composables/useImportDemos";
import { useFiveEImport } from "@/features/import/composables/useFiveEImport";
import DeathNoticeLine from "@/features/clips/components/DeathNoticeLine.vue";
```

## Dependency Direction

```
app/ (shell, router)
  в†‘
features/* (pages depend on composables)
  в†‘
shared/ (types, i18n, state)
```

- `shared/` never depends on `features/` or `app/`
- `features/*` never depend on each other's pages (only shared state via composables or `wailsjs`)
- `composables/` are the only way to share state between components within a feature
- Cross-feature state sharing goes through `shared/state/` or `wailsjs` backend calls
