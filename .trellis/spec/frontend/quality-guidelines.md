# Quality Guidelines

> Code quality, build verification, and forbidden patterns for the frontend.

---

## Build & Verification Commands

```bash
# Full build with type-check (MANDATORY before any frontend commit)
cd frontend && npm run build

# This runs: vue-tsc --noEmit && vite build
```

The build command **must pass** before committing any frontend changes. `vue-tsc` performs full-project TypeScript type checking — type errors are build errors.

---

## Code Style

### TypeScript

- **TypeScript 4.6+** with strict mode (inferred from tsconfig)
- Use **explicit return types** for composable functions (inferred types are acceptable but explicit is preferred)
- Use **type imports** with `import type { ... }` syntax:

```ts
import type { DemoClipKill, DemoListEntry } from "@/shared/types";
import { computed, ref } from "vue";
```

- Use **`as` type assertions sparingly** — prefer type-safe conditional checks
- No `any` type — use `unknown` if the type is truly unknown, then narrow with type guards

### Vue

- All components use `<script setup lang="ts">`
- No Options API (`data()`, `methods: {}`, `computed: {}`)
- Scoped styles only (`<style scoped>`)
- Template expressions should be simple — complex logic goes in `computed` or methods

## Auto-Imports

The project uses two unplugin plugins that auto-register:

### unplugin-auto-import

Vue Composition API functions are auto-imported at build time, but **must still be explicitly imported** in every `<script setup>` block that uses them. The build-time injection handles runtime availability, but global type declarations may not resolve reliably on Windows TypeScript strict builds. See [Component Guidelines](./component-guidelines.md#vue-api-import-principle) for details.

```ts
// ✅ Always explicitly import Vue APIs
import { computed, nextTick, onBeforeUnmount, onMounted, ref, watch } from "vue";

// ❌ Do NOT rely on unplugin-auto-import globals without explicit imports
// (May cause TS2304 on Windows)
```

### unplugin-vue-components

Naive UI components are auto-registered:

```vue
<!-- ✅ These work without manual import -->
<n-card />
<n-button />
<n-tag />
<n-space />
<n-collapse />
<!-- ❌ Do NOT add: import { NCard, NButton } from "naive-ui" -->
```

## Wails Go/JS Binding

### Auto-generated Files (NEVER hand-edit)

```bash
frontend/wailsjs/**          # Go↔JS bindings — regenerated by wails build/dev
frontend/src/auto-imports.d.ts  # Auto-generated by unplugin-auto-import
frontend/src/components.d.ts    # Auto-generated by unplugin-vue-components
```

If you need to add a new backend method:
1. Add the Go method on `internal/app.App`
2. Run `wails dev` (or `wails build`) to regenerate `frontend/wailsjs/`
3. The new method is available via `window.go.app.App.NewMethod()`

## Frontend-Specific Forbidden Patterns

| Pattern | Why | Instead |
|---------|-----|---------|
| `window.go.app.App.xxx()` directly | No error handling if API not loaded | `callBackend("xxx", args)` helper |
| `import { NCard } from "naive-ui"` | Auto-registered by unplugin | Use `<n-card>` directly |
| `import { ref, computed } from "vue"` | ❌ DO import — see [Vue API Import Principle](./component-guidelines.md#vue-api-import-principle) | Not doing so causes TS2304 on Windows |
| `export default { data() {} }` | Options API, deprecated pattern | `<script setup lang="ts">` |
| `any` type annotation | Loses type safety | `unknown` + type guard |
| State inside `export function useX()` | Not shared, recreated per call | Module-level `ref()` |
| Pinia or Vuex install | Overkill for this project's needs | Module-level composable singletons |
| `vue-i18n` install | Custom system already exists | Use `@/shared/i18n` |
| Editing `en-US.json` | User-maintained file | Only edit `zh-CN.json` |
| Editing `wailsjs/` files | Auto-generated, will be overwritten | Change Go binding, regenerate |

## Testing

The frontend currently has **no test infrastructure** — no unit tests, no component tests, no E2E tests. Quality is verified through:

1. **Type-check** — `vue-tsc --noEmit` (part of `npm run build`)
2. **Build** — `vite build` succeeds
3. **Manual testing** — run `wails dev` and verify in the desktop app

If adding tests in the future:
- Use **Vitest** (Vite-native test runner) — do not install Jest
- Place test files alongside source files: `*.test.ts` or `*.spec.ts`
- Use `@vue/test-utils` for component tests

## Code Review Checklist

- [ ] `cd frontend && npm run build` passes (type-check + vite build)
- [ ] No hand-edited auto-generated files (`wailsjs/`, `auto-imports.d.ts`, `components.d.ts`)
- [ ] All user-facing strings use `t()` from `@/shared/i18n`
- [ ] All backend calls go through `callBackend()` helper (not raw `window.go.app.App.xxx()`)
- [ ] All Vue APIs (`ref`, `computed`, `nextTick`, etc.) are explicitly imported from `"vue"` — do not rely on unplugin-auto-import globals
- [ ] No manual imports of Naive UI components — let unplugin-vue-components handle it
- [ ] State is module-level (outside `export function`) for shared composables
- [ ] Template expressions are simple — complex logic is in computed/methods
- [ ] Styles use `scoped` — no global CSS pollution
- [ ] New types added to `shared/types.ts` if they cross feature boundaries
- [ ] New translation keys added to `zh-CN.json` (not `en-US.json`)
